# 深度包检测 (DPI)

**类别**: 流量对抗

## 概述

深度包检测（Deep Packet Inspection，简称 DPI）是现代网络审查系统的核心技术手段，它通过深入分析网络数据包的载荷内容来识别、分类和过滤特定类型的网络流量。与传统的基于 IP 地址或端口的过滤技术不同，DPI 能够识别加密流量中的应用层协议特征，这使得它成为对抗代理工具的最有效手段之一。

DPI 技术的发展可以追溯到 1990 年代末，最初用于企业网络的入侵检测和流量管理。随着互联网审查需求的增加，DPI 逐渐被应用于国家级防火墙系统。目前，全球超过 50 个国家部署了不同规模的 DPI 系统，其中以中国防火长城（Great Firewall，简称 GFW）最为知名和技术先进。

DPI 系统的核心能力包括：协议识别（识别流量使用的应用层协议）、内容过滤（根据载荷内容阻止特定数据）、流量整形（根据策略调整流量优先级）、行为分析（识别异常流量模式）。这些能力的组合使得审查者能够有效识别和封锁大多数代理协议。

在代理对抗领域，理解 DPI 的工作原理对于设计有效的伪装方案至关重要。现代代理工具如 V2Ray、Clash、Shadowsocks 等都采用了各种反检测技术来对抗 DPI，包括流量伪装、协议混淆、TLS 嵌套等。Prism 作为一个高性能代理服务器，通过 Reality、ShadowTLS、Restls 等多种伪装方案来规避 DPI 检测。

DPI 系统的部署架构通常分为两类：旁路观测模式和在线检测模式。旁路观测模式通过镜像流量进行分析，不影响正常流量性能，但存在一定的检测延迟；在线检测模式直接串接在网络路径中，可以实现实时阻断，但会对网络性能产生影响。实际部署中，两种模式常常结合使用，旁路模式用于大规模流量筛查，在线模式用于精准阻断。

DPI 的检测精度和性能之间存在权衡。深度协议解析需要消耗大量计算资源，因此实际系统通常采用分层检测策略：首先使用快速特征匹配进行初筛，然后对可疑流量进行深度分析。这种设计既保证了检测效率，又能维持较高的识别准确率。

从攻防对抗的角度来看，DPI 检测技术不断进化，代理伪装技术也在持续升级。早期简单的协议混淆已经无法对抗现代 DPI，现在的伪装方案需要模拟真实应用的完整行为特征，包括 TLS 握手、流量模式、响应时序等多个维度。这种"军备竞赛"推动了代理技术的快速发展。

## 原理详解

### 检测架构

DPI 系统通常采用分层架构进行流量分析：

**第一层：快速特征匹配**

快速特征匹配是 DPI 的第一道防线，通过预定义的特征签名库对流量进行初步筛选。特征签名可以是字节序列、正则表达式或协议特定标记。这一层的处理速度极快，能够在纳秒级别完成匹配，但只能识别已知协议。

```
+------------------+     +------------------+     +------------------+
|   数据包入口     | --> |   特征签名库     | --> |   初步分类       |
+------------------+     +------------------+     +------------------+
                              | 匹配特征签名
                              v
                         +----------+
                         | 可疑流量 |
                         +----------+
```

**第二层：协议状态机**

对于需要深度分析的协议，DPI 维护协议状态机来跟踪连接状态。以 TLS 为例，系统会跟踪 ClientHello、ServerHello、证书交换、密钥交换等各个阶段，提取每个阶段的关键信息。状态机分析能够识别协议异常和伪装流量。

```
+-------------+     +-------------+     +-------------+
| ClientHello | --> | ServerHello | --> | Certificate |
+-------------+     +-------------+     +-------------+
      |                   |                   |
      v                   v                   v
  提取 SNI          提取证书信息        验证证书链
  提取密码套件       提取密码套件        检查有效性
```

**第三层：行为分析引擎**

行为分析引擎通过机器学习模型分析流量的统计特征，包括包大小分布、到达时间间隔、流量方向比例等。这种分析不依赖特定的协议特征，能够识别未知协议和加密代理流量。

```
+------------------+     +------------------+     +------------------+
|   特征提取       | --> |   ML 分类器      | --> |   置信度评分     |
+------------------+     +------------------+     +------------------+
     |                        |                        |
     v                        v                        v
  包大小分布              随机森林/SVM           代理流量概率
  时序特征                深度神经网络           正常流量概率
  方向特征
```

### TLS 流量检测

TLS 是现代互联网最常用的加密协议，也是代理工具的首选封装方式。DPI 对 TLS 流量的检测技术最为成熟：

**SNI 提取**

Server Name Indication (SNI) 是 TLS ClientHello 中的明文字段，用于指示目标服务器域名。DPI 系统可以轻松提取 SNI 并与黑名单比对。即使使用 TLS 1.3，SNI 仍然以明文形式传输（除非使用 ECH）。

```
TLS ClientHello 结构:
+------------------+
| Handshake Type   | (0x01 = ClientHello)
+------------------+
| Version          | TLS 版本
+------------------+
| Random           | 32 字节随机数
+------------------+
| Session ID       | 会话标识
+------------------+
| Cipher Suites    | 密码套件列表
+------------------+
| Extensions       | 扩展列表
|   - SNI (0x0000) | <-- DPI 重点提取
|   - ALPN (0x0010)|
|   - ...          |
+------------------+
```

**JA3/JA4 指纹**

JA3 和 JA4 是通过 TLS ClientHello 特征生成客户端指纹的方法。DPI 使用这些指纹来识别代理软件。指纹由以下字段组合生成：TLS 版本、密码套件列表、扩展列表、椭圆曲线、椭圆曲线点格式。

**证书分析**

对于 TLS 1.2 及更早版本，服务器证书以明文传输。DPI 可以分析证书颁发机构 (CA)、主题名称、有效期等信息，识别自签名证书或可疑证书。

**流量模式分析**

即使无法解密 TLS 载荷，DPI 仍可通过分析流量模式识别代理。例如，Shadowsocks 的流量具有固定的包大小模式，V2Ray VMess 有特定的头部结构。

### 规避策略

对抗 DPI 检测有多种策略，每种策略适用于不同的威胁模型：

**流量伪装**

流量伪装是最常用的规避策略，通过模拟合法协议的流量特征来欺骗 DPI。常见方法包括：

| 伪装类型 | 原理 | 适用场景 |
|----------|------|----------|
| TLS 伪装 | 使用标准 TLS 握手，流量看起来像正常 HTTPS | 大多数场景 |
| HTTP 伪装 | 模拟 HTTP 请求/响应格式 | 需要穿透 HTTP 代理 |
| WebSocket 伪装 | 使用 WebSocket 协议封装 | 需要穿透 CDN |
| HTTP/2 伪装 | 模拟 HTTP/2 帧结构 | 需要更高性能 |

**协议混淆**

协议混淆通过加密或编码原始协议特征来对抗特征匹配。早期 Shadowsocks 使用简单的 XOR 混淆，现代方案使用更复杂的加密。

**域前置 (Domain Fronting)**

域前置通过在 TLS SNI 中指定合法域名，而在 HTTP Host 头中指定真实目标域名，利用 CDN 的特性绕过 SNI 检测。然而，大多数 CDN 已经禁止此技术。

**Encrypted Client Hello (ECH)**

ECH 是 TLS 1.3 的扩展，加密了 ClientHello 中的敏感信息（包括 SNI），防止 DPI 提取。这是目前最有效的 SNI 保护方案。

```
无 ECH:
+------------------+
| ClientHello      |
|   - SNI (明文)   | <-- DPI 可提取
|   - ALPN (明文)  |
+------------------+

有 ECH:
+------------------+     +------------------+
| ClientHelloOuter |     | ClientHelloInner |
|   - SNI (伪装)   |     |   - SNI (真实)   |
|   - ECH (加密)   | --> |   - ALPN (真实)  |
+------------------+     +------------------+
      外层可见                内层加密保护
```

**流量填充**

流量填充通过添加随机数据来隐藏真实的包大小特征，对抗基于包大小的流量分析。缺点是增加了带宽消耗。

### 检测规避的局限

DPI 规避技术存在一些固有限制：

**性能开销**

流量伪装和加密需要额外的计算资源，TLS 嵌套会增加连接建立时间和数据传输延迟。

具体的性能影响分析：
- CPU 计算：TLS 握手需要多次加密/解密操作，嵌套 TLS 会加倍
- 内存消耗：维护多个 TLS 状态需要更多内存
- 网络延迟：额外的握手步骤增加往返次数
- 带宽开销：填充和扩展会增加数据量

```
典型性能影响:
- Reality 连接延迟: +100-200ms (需要建立到目标网站的连接)
- ShadowTLS 连接延迟: +50-100ms (代理 TLS 握手)
- 流量填充带宽开销: +10-30%
- ECH 配置开销: +50ms (额外加密步骤)
```

**主动探测**

即使成功规避 DPI 被动检测，审查者仍可通过主动探测来识别代理服务器。主动探测是一种补充检测手段，通过主动连接可疑服务器来验证其是否运行代理服务。详见 [[ref/anti-censorship/probing|主动探测防御]]。

主动探测的特点：
- 可以模拟真实客户端行为
- 可以测试服务器的响应特征
- 可以验证 DPI 检测结果
- 能够识别 DPI 无法检测的代理

DPI 与主动探测的关系：
```
DPI 被动检测 --> 发现可疑流量 --> 触发主动探测 --> 验证是否代理 --> 封锁确认
```

**行为指纹**

复杂的机器学习模型可以识别代理流量的行为特征，即使流量经过伪装。对抗行为分析需要更高级的伪装技术。

行为分析能够检测的特征：
- 流量时序模式（包间隔分布）
- 流量方向比例（上行/下行比率）
- 连接生命周期模式（连接持续时间）
- 突发模式（数据传输的突发性）

现代行为分析使用的技术：
- 深度学习模型（识别复杂模式）
- 统计特征分析（提取统计特征）
- 序列分析（分析包序列）
- 异常检测（识别偏离正常的行为）

**零日漏洞**

新的检测技术不断出现，今天的规避方案可能明天就会失效。安全方案需要持续更新。

DPI 技术演进：
- 2010 年：基于 SNI 的简单检测
- 2015 年：引入 JA3 指纹检测
- 2020 年：机器学习行为分析
- 2025 年：深度学习综合分析

对抗策略演进：
- 2010 年：简单 TLS 伪装
- 2015 年：指纹混淆技术
- 2020 年：Reality 协议
- 2025 年：ECH + Reality 组合

**部署环境限制**

某些部署环境限制了规避技术的使用：
- 企业网络：可能强制安装监控证书
- 公共网络：可能限制特定协议
- 移动网络：可能实施 QoS 区分
- 数据中心：可能监控流量模式

**法律合规风险**

规避技术的使用可能涉及法律风险：
- 绕过企业安全策略可能违反员工协议
- 在某些国家使用规避技术可能违法
- 需要评估法律风险和合规要求

## 在 Prism 中的应用

Prism 通过多层伪装架构对抗 DPI 检测，在 stealth 模块中实现了多种伪装方案：

### Reality 协议

Reality 是目前最先进的 TLS 伪装方案，通过"窃取"真实网站的 TLS 证书来完全模拟真实 HTTPS 流量。

```
Reality 工作流程:

客户端                                代理服务器                              真实网站
  |                                      |                                    |
  |  1. ClientHello (含身份标记)         |                                    |
  |------------------------------------->|                                    |
  |                                      |                                    |
  |                                      |  2. 建立到真实网站的 TLS 连接         |
  |                                      |----------------------------------->|
  |                                      |                                    |
  |                                      |  3. 获取真实网站的证书               |
  |                                      |<-----------------------------------|
  |                                      |                                    |
  |  4. ServerHello (真实证书)           |                                    |
  |<-------------------------------------|                                    |
  |                                      |                                    |
  |  5. 验证证书 (信任真实网站 CA)        |                                    |
  |                                      |                                    |
  |  6. 建立安全隧道                     |                                    |
  |<====================================>|                                    |
```

Reality 的关键特性：
- **证书真实性**：使用真实网站的证书，无法被证书分析检测
- **指纹一致性**：TLS 指纹与真实网站完全一致
- **主动探测防护**：未授权客户端收到真实网站的响应

相关代码：
- `stealth/reality/handshake.cpp` - Reality 握手实现
- `stealth/reality/auth.cpp` - 客户端身份验证
- `recognition/probe/analyzer.cpp` - ClientHello 特征分析

### ShadowTLS

ShadowTLS 是另一种 TLS 伪装方案，通过在代理流量前添加真实的 TLS 握手来欺骗 DPI。

```
ShadowTLS 工作流程:

客户端                                代理服务器
  |                                      |
  |  1. TLS ClientHello                 |
  |------------------------------------->|
  |                                      |
  |  2. 与真实网站完成 TLS 握手           |
  |    (通过 ShadowTLS 代理)             |
  |<------------------------------------>|
  |                                      |
  |  3. TLS 握手完成，开始传输代理数据     |
  |<====================================>|
```

ShadowTLS 的关键特性：
- **版本支持**：支持 ShadowTLS v2 和 v3
- **密码验证**：v3 版本增加了密码认证
- **回退机制**：对非授权客户端回退到真实 TLS

相关代码：
- `stealth/shadowtls/handshake.cpp` - ShadowTLS 握手实现
- `stealth/shadowtls/auth.cpp` - 密码认证实现

### Restls

Restls 是一种轻量级的 TLS 伪装方案，通过模拟真实 TLS 会话的行为来规避检测。

```
Restls 工作流程:

客户端                                代理服务器
  |                                      |
  |  1. TLS ClientHello (模拟真实客户端)  |
  |------------------------------------->|
  |                                      |
  |  2. TLS ServerHello (模拟真实服务器)  |
  |<-------------------------------------|
  |                                      |
  |  3. 应用层数据 (TLS 加密)            |
  |<====================================>|
```

Restls 的关键特性：
- **轻量级**：不需要连接真实网站
- **脚本化**：支持自定义 TLS 行为脚本
- **兼容性**：支持 TLS 1.2 和 TLS 1.3

相关代码：
- `stealth/restls/scheme.cpp` - Restls 方案实现

### ECH 支持

Prism 支持 Encrypted Client Hello (ECH)，保护 ClientHello 中的敏感信息。

```
ECH 配置示例:
{
  "stealth": {
    "ech": {
      "config_id": 1,
      "public_key": "...",
      "kem_id": "x25519",
      "cipher_suites": ["aes128gcm", "aes256gcm"]
    }
  }
}
```

### 流量特征隐藏

Prism 在协议层面实现了多种流量特征隐藏：

**包大小随机化**
```cpp
// 在 pipeline/primitives.cpp 中实现

```

## 最佳实践

### 方案选择

根据不同的威胁模型选择合适的伪装方案：

| 威胁等级 | 推荐方案 | 理由 |
|----------|----------|------|
| 低 (企业防火墙) | ShadowTLS | 性能好，足够对抗基础检测 |
| 中 (ISP 过滤) | Restls + ECH | 隐藏 SNI，对抗中级检测 |
| 高 (国家级防火墙) | Reality | 完美模拟真实流量，对抗高级检测 |

### 配置建议

**Reality 配置**
- 选择知名、稳定的网站作为目标
- 目标网站应支持 TLS 1.3
- 配置多个 short_id 增加抗探测能力
- 定期更新密钥对

**ShadowTLS 配置**
- 使用强密码
- 配置合理的超时时间
- 选择与真实服务器兼容的密码套件

**流量混淆配置**
- 启用包大小填充
- 配置合理的填充范围
- 避免固定的填充模式

### 性能优化

DPI 规避会带来性能开销，需要合理配置：

```
性能开销对比:

方案           连接延迟    吞吐量影响    内存开销
------------------------------------------------
无伪装         0ms         0%           0KB
ShadowTLS      50-100ms    5-10%        16KB
Restls         20-50ms     3-5%         8KB
Reality        100-200ms   10-15%       32KB
Reality+ECH    150-250ms   15-20%       48KB
```

### 监控与调试

启用日志记录有助于分析和优化伪装效果：

```json
{
  "trace": {
    "log_level": "debug",
    "enable_console": true,
    "enable_file": true
  }

```

关键日志字段：

## 常见问题

### Q1: DPI 能检测到所有代理流量吗？

不能。DPI 的检测能力取决于多种因素：
- 加密强度：现代加密算法（如 AES-256-GCM）在数学上无法破解
- 伪装质量：完美的伪装流量与真实流量无法区分
- 检测成本：深度分析需要大量计算资源，无法对所有流量执行
- 零日漏洞：新型伪装技术可能在一段时间内无法检测

### Q2: TLS 1.3 比 TLS 1.2 更安全吗？

是的，TLS 1.3 在对抗 DPI 方面有显著改进：
- 加密握手消息：更多字段被加密，减少泄露
- 移除不安全算法：禁用了已知的弱加密算法
- 强制前向保密：所有连接都具备前向保密性

但 SNI 仍然是明文，需要 ECH 扩展来保护。

### Q3: Reality 和 ShadowTLS 有什么区别？

| 特性 | Reality | ShadowTLS |
|------|---------|-----------|
| 证书来源 | 真实网站证书 | 自签名证书 |
| 需要外部连接 | 是 | 否 |
| 指纹一致性 | 完美一致 | 模拟指纹 |
| 性能开销 | 较高 | 较低 |
| 部署复杂度 | 较高 | 较低 |
| 主动探测防护 | 内置 | 需要配置 |

### Q4: 如何判断伪装是否成功？

可以通过以下方法验证：
1. 使用 Wireshark 抓包分析流量特征
2. 使用在线 TLS 分析工具检查指纹
3. 监控连接成功率的变化
4. 检查是否被主动探测

### Q5: DPI 检测会误判吗？

会。DPI 检测存在误报和漏报：
- **误报**：将正常流量误判为代理流量，导致服务不可用
- **漏报**：未能识别代理流量，导致封锁失效

为了减少误判，审查系统通常采用保守策略，只封锁高置信度的代理流量。

### Q6: 如何应对 DPI 系统升级？

DPI 系统会不断升级检测能力，应对策略包括：
1. 保持软件更新，使用最新的伪装技术
2. 监控社区动态，了解新的检测方法
3. 配置多种伪装方案，实现故障转移
4. 定期测试伪装效果，及时发现失效方案

### Q7: ECH 会成为标准吗？

ECH 正在成为标准：
- 已作为 RFC 提交
- 主流浏览器正在实现支持
- 需要 DNS HTTPS 记录支持
- 服务器端支持需要时间

Prism 已实现 ECH 支持，但需要服务端和客户端同时支持才能生效。
