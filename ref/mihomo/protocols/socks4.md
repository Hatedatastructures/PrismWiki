# SOCKS4 协议

SOCKS4 是 SOCKS（Socket Secure）协议的早期版本，定义于 1990 年代。它提供了一个简单的 TCP 代理机制，仅支持基于 IP 地址的连接转发，不支持认证和 UDP 代理。SOCKS4a 是其扩展版本，增加了域名传递能力。

## 协议概述

SOCKS4 协议的核心特性：

- **仅支持 TCP 代理**：不提供 UDP 关联机制
- **无认证机制**：不支持用户名/密码验证，仅传输用户标识符（User ID）
- **SOCKS4a 扩展**：允许客户端传递目标域名而非 IP 地址
- **不支持 IPv6**：协议设计中仅包含 32 位 IPv4 地址字段
- **简单高效**：协议头部仅 8 字节，开销极小

SOCKS4 是 SOCKS 协议家族的第二个版本（SOCKS4 实际上是 SOCKS 协议的第一个公开发布版本，之前的 SOCKS 版本未公开）。相比 SOCKS5，它功能有限但实现简单，在某些遗留系统中仍然使用。

## SOCKS4 帧格式

### 请求格式

SOCKS4 请求的帧格式固定为 8 字节头部加上变长的用户 ID 字段：

```
+-----+------+----------+----------+-----------------+
| VER | CMD  | DST.PORT | DST.IP   | USER ID         |
+-----+------+----------+----------+-----------------+
| 1 B | 1 B  | 2 B      | 4 B      | 可变长 + 1 B 0x00|
+-----+------+----------+----------+-----------------+
```

| 字段 | 大小 | 说明 |
|------|------|------|
| `VER` | 1 字节 | 协议版本，固定为 `0x04` |
| `CMD` | 1 字节 | 命令码：`0x01` = CONNECT（TCP 连接），`0x02` = BIND（监听） |
| `DST.PORT` | 2 字节 | 目标端口（网络字节序，大端） |
| `DST.IP` | 4 字节 | 目标 IP 地址（网络字节序） |
| `USER ID` | 可变长 | 用户标识符，以 `0x00` 字节结尾 |

### 响应格式

服务器响应格式固定为 8 字节：

```
+-----+------+----------+----------+
| VN  | CODE | DST.PORT | DST.IP   |
+-----+------+----------+----------+
| 1 B | 1 B  | 2 B      | 4 B      |
+-----+------+----------+----------+
```

| 字段 | 大小 | 说明 |
|------|------|------|
| `VN` | 1 字节 | 响应版本，固定为 `0x00` |
| `CODE` | 1 字节 | 响应码：`0x5A` = 成功，`0x5B` = 失败/拒绝，`0x5C` = 用户 ID 无效，`0x5D` = 用户 ID 格式错误 |
| `DST.PORT` | 2 字节 | 绑定的端口（BIND 模式下有效） |
| `DST.IP` | 4 字节 | 绑定的 IP（BIND 模式下有效） |

### CONNECT 命令流程

```
Client                 SOCKS4 Server
  |  VER=0x04, CMD=0x01   |
  |  DST.PORT, DST.IP     |
  |  USER ID + 0x00       |
  |--------------------->|
  |                       | 解析请求
  |                       | 建立到目标的 TCP 连接
  |  VN=0x00, CODE=0x5A   |
  |<---------------------|
  |  TCP 数据透传         |
```

### BIND 命令流程

BIND 命令用于 FTP 等需要反向连接的场景：

```
Client                 SOCKS4 Server
  |  VER=0x04, CMD=0x02   |
  |  DST.PORT, DST.IP     |
  |  USER ID + 0x00       |
  |--------------------->|
  |                       | 开始监听端口
  |  VN=0x00, CODE=0x5A   |
  |  (BND.PORT, BND.IP)   |
  |<---------------------|
  |                       | 等待第三方连接
  |                       | 第三方连接到达
  |  VN=0x00, CODE=0x5A   |
  |  (DST.PORT, DST.IP)   |
  |<---------------------|
  |  TCP 数据透传         |
```

## SOCKS4a 扩展（域名解析）

### 问题背景

原始 SOCKS4 协议要求客户端在发送请求前解析目标域名为 IP 地址。这有两个问题：

1. 客户端需要有 DNS 解析能力
2. 无法代理 DNS 查询本身
3. 某些场景下客户端无法解析域名（如 DNS 被封锁）

### SOCKS4a 解决方案

SOCKS4a 通过以下机制解决域名传递问题：

- 当目标 IP 地址为 `0.0.0.x`（x > 0，即最高字节为 0，其余非零）时，表示这是一个 SOCKS4a 请求
- 在 USER ID 字段之后追加目标域名，以 `0x00` 字节结尾

```
+-----+------+----------+----------------+-----------+----------+------+------+
| VER | CMD  | DST.PORT | DST.IP(0.0.0.x)| USER ID   | 0x00   | HOST | 0x00 |
+-----+------+----------+----------------+-----------+----------+------+------+
| 1 B | 1 B  | 2 B      | 4 B            | 可变长    | 1 B    | 可变  | 1 B  |
+-----+------+----------+----------------+-----------+----------+------+------+
```

SOCKS4a 的巧妙之处在于复用了 DST.IP 字段的特殊值作为域名传递的标志，保持了协议的向后兼容性。

### SOCKS4a 请求示例

连接 `example.com:80`，用户 ID 为 `user`：

```
04 01          # VER=4, CMD=CONNECT
00 50          # DST.PORT=80
00 00 00 01    # DST.IP=0.0.0.1（SOCKS4a 标志）
75 73 65 72 00 # USER ID="user" + 0x00
65 78 61 6D 70 6C 65 2E 63 6F 6D 00  # HOST="example.com" + 0x00
```

## 与 SOCKS5 的区别

### 全面对比

| 特性 | SOCKS4 | SOCKS4a | SOCKS5 |
|------|--------|---------|--------|
| 发布年份 | 1990s | 1990s | 1996 (RFC 1928) |
| TCP 代理 | 支持 | 支持 | 支持 |
| UDP 代理 | 不支持 | 不支持 | 支持（UDP Associate） |
| 认证 | 无 | 无 | 支持（用户名/密码、GSSAPI 等） |
| 域名传递 | 不支持 | 支持 | 原生支持 |
| IPv6 | 不支持 | 不支持 | 支持 |
| BIND 命令 | 支持 | 支持 | 支持 |
| 地址类型 | 仅 IPv4 | 仅 IPv4 | IPv4 / IPv6 / 域名 |
| 请求头部大小 | 8 + 变长 | 8 + 变长 | 可变（4-262 字节） |
| 协议复杂度 | 低 | 低 | 中 |

### SOCKS5 地址类型对比

SOCKS5 在地址类型字段中明确指定地址类型：

```
SOCKS5 请求格式：
+-----+------+----------+
| VER | CMD  | RSV      |
+-----+------+----------+

+----------+----------+----------+
| ATYP     | DST.ADDR | DST.PORT |
+----------+----------+----------+

ATYP 值：
0x01 = IPv4（4 字节）
0x03 = 域名（1 字节长度 + 变长域名）
0x04 = IPv6（16 字节）
```

相比之下，SOCKS4 硬编码为 IPv4 地址，SOCKS4a 使用特殊 IP 值来标志域名传递。

### 认证对比

SOCKS4/4a 完全没有认证机制。任何知道 SOCKS 服务器地址和端口的客户端都可以使用代理服务。

SOCKS5 支持多种认证方法：
- `0x00`：无认证
- `0x01`：GSSAPI
- `0x02`：用户名/密码
- `0x03-0x7E`：IANA 分配
- `0x7F-0xFE`：私有方法
- `0xFF`：无可接受的方法

## 安全性考量

SOCKS4 协议存在以下安全问题：

1. **无认证**：任何人都可以使用代理服务，如果服务器暴露到公网，可能被滥用
2. **无加密**：所有数据（包括 USER ID）以明文传输
3. **无完整性保护**：数据可能被中间人篡改
4. **IPv4 限制**：无法代理到 IPv6 目标

因此，SOCKS4 仅推荐在受信任的局域网内使用。对于公网代理，应使用 SOCKS5 并启用认证。

## mihomo 实现位置

| 文件 | 描述 |
|------|------|
| `adapter/outbound/base.go` | SOCKS4 通过基础适配器处理 |
| `transport/socks4/socks4.go` | SOCKS4/4a 协议实现，包含请求构建和响应解析 |
| `transport/socks4/socks4a.go` | SOCKS4a 域名传递逻辑 |

mihomo 的 SOCKS4 实现主要作为一个出站代理使用，即 mihomo 作为客户端连接到远程 SOCKS4 服务器。实现包括：

1. **请求构建**：根据目标地址类型（IP 或域名）构建 SOCKS4 或 SOCKS4a 请求
2. **响应解析**：解析服务器的 8 字节响应，判断连接是否成功
3. **域名自动检测**：如果目标是域名而非 IP，自动使用 SOCKS4a 格式

## 与 Prism 的兼容性

| Prism 模块 | 兼容性 | 说明 |
|------------|--------|------|
| Agent Worker | 完全兼容 | 基础 TCP 代理 |
| Pipeline | 完全兼容 | 仅 TCP 流 |
| UDP 支持 | 不兼容 | SOCKS4 不支持 UDP |
| IPv6 | 不兼容 | 协议限制 |

## YAML 配置示例

### 基本配置

```yaml
proxies:
  - name: "socks4-proxy"
    type: socks4
    server: server.example.com
    port: 1080
```

### 连接流程

```yaml
# mihomo 作为 SOCKS4 客户端
proxies:
  - name: "socks4-outbound"
    type: socks4
    server: 192.168.1.100
    port: 1080

rules:
  - DOMAIN-SUFFIX,example.com,socks4-outbound
```

当流量匹配到该规则时，mihomo 会：
1. 连接到 `192.168.1.100:1080`
2. 发送 SOCKS4/4a 请求，目标为 `example.com` 的解析 IP
3. 收到成功响应后，开始 TCP 数据透传
