# SHA224 哈希工具

SHA224 是 Trojan 协议专用的 header-only inline 哈希工具，提供 SHA-224 哈希计算和凭据标准化功能。

> 源码: [[include/prism/crypto/sha224.hpp]]

## 设计决策

### 为什么 Trojan 用 SHA-224 而非 SHA-256？

Trojan 协议规范要求将用户密码以 SHA-224 哈希后的 56 字节十六进制字符串形式存储和传输。SHA-224 是 SHA-256 的截断版本，输出 28 字节（56 hex chars），比 SHA-256 的 32 字节更短，在 Trojan 协议头中节省空间。

**后果**: SHA-224 的安全强度为 112 位（低于 SHA-256 的 128 位），但对于凭据哈希而言足够。Prism 中仅 Trojan 使用 SHA-224，其他模块使用 BLAKE3 或 HKDF-SHA256。

### 为什么提供 `normalize_credential` 而非让调用方自行处理？

Trojan 配置文件中的密码字段有两种合法格式：明文密码（如 `"my-secret"`）或已经 SHA-224 哈希的 56 字节十六进制字符串。`normalize_credential` 自动检测并统一处理，避免配置加载代码重复。

**后果**: 配置文件中两种格式都合法，降低运维出错概率。但修改此函数的判断逻辑会导致配置兼容性断裂。

### 为什么返回 `memory::string` 而非 `std::string`？

遵循 Prism 的 PMR 内存策略，使用全局池分配器。SHA-224 在配置加载时调用（冷路径），但保持一致的内存策略比路径分类更重要。

**后果**: 与其他 crypto 模块的返回类型风格一致。

## 约束

### 输出固定 56 字节

**类型**: 资源上限

**规则**: `sha224()` 输出恒为 56 字节十六进制字符串（28 字节哈希 * 2 hex chars/byte）

**违反后果**: 不可违反——SHA-224 算法输出长度固定

**源码依据**: `sha224.hpp:27-42`（遍历 28 字节哈希，每个字节输出 2 hex chars）

### `normalize_credential` 的判断逻辑

**类型**: 调用顺序

**规则**: 输入恰好 56 字节且全部为 `[0-9a-fA-F]` 字符时，视为已哈希，直接返回；否则计算 SHA-224

**违反后果**: 如果用户的明文密码恰好是 56 字节纯十六进制字符串，会被误判为已哈希

**源码依据**: `sha224.hpp:69-77`

### 依赖 OpenSSL SHA224

**类型**: 生命周期

**规则**: `sha224.hpp` 包含 `<openssl/sha.h>`，调用 `SHA224()` 函数。虽然 Trojan 场景在启动时调用，但 OpenSSL 初始化必须已完成

**违反后果**: 若 OpenSSL 未初始化，`SHA224()` 行为未定义

**源码依据**: `sha224.hpp:11`

## 函数详解

### SHA-224 哈希（56 字节输出）

**签名**: `auto sha224(string_view input) -> memory::string`

计算输入字符串的 SHA-224 哈希值，返回 56 字节的十六进制字符串。

**特点**: 使用 OpenSSL 的 `SHA224()` 函数，栈上分配 `SHA224_DIGEST_LENGTH`（28 字节）缓冲区，零堆分配。

### `is_hex` 辅助函数

**签名**: `auto is_hex(string_view str) -> bool`

检查字符串是否全部为十六进制字符 `[0-9a-fA-F]`。

### `normalize_credential` 凭据标准化

**签名**: `auto normalize_credential(string_view credential) -> memory::string`

Trojan 凭据标准化：如果输入已经是 56 字节 hex 字符串则直接返回，否则计算 SHA-224。

> 源码: `include/prism/crypto/sha224.hpp:69`

```cpp
credential.size() == 56 && is_hex(credential)?
    return string(credential)        // 已哈希
    : return sha224(credential)      // 明文哈希
```

**调用方**: [[core/loader/load|loader::load]] 配置加载时调用，将用户密码标准化后存入 account entry

## Trojan 凭据流程

```
配置文件:
  password: "my-secret-password"

处理:
  normalize_credential("my-secret-password")
  => sha224("my-secret-password")
  => "a1b2c3d4e5f6..." (56 hex chars)
  => 存入 account::entry

验证时:
  Trojan 协议头中的 SHA-224 哈希与 account entry 比对
```

**安全说明**: Trojan 协议使用 SHA-224 单向哈希（无盐），不支持 BLAKE3 或 HKDF-SHA256 的密钥派生。

## 故障场景

### 明文密码被误判为已哈希

**触发条件**: 用户明文密码恰好为 56 字节且全部为 `[0-9a-f]` 字符

**传播路径**: `normalize_credential` 直接返回明文 -> 存入 account entry -> Trojan 协议头中的 SHA-224 哈希与明文不匹配 -> 认证失败

**外部表现**: Trojan 用户无法认证

**恢复机制**: 修改密码或使用带非十六进制字符的密码

**日志关键字**: 无直接日志（认证静默失败）

### OpenSSL 未初始化

**触发条件**: 在 `memory::system::enable_global_pooling()` 或 OpenSSL 初始化之前调用 `sha224()`

**传播路径**: `SHA224()` 行为未定义 -> 可能返回错误哈希或崩溃

**外部表现**: 进程崩溃或 Trojan 认证失败

**恢复机制**: 确保启动顺序正确

**日志关键字**: 无
