# GFW 封锁原理深度解析

Great Firewall（GFW）是中国的互联网审查系统，通过多种技术手段检测和封锁翻墙流量。理解 GFW 的工作原理是设计抗审查代理协议的基础。

## 概述

### GFW 的定位与架构

GFW 是国家级互联网审查系统，部署在网络边界：

```
中国境内网络                         互联网边界                        全球互联网
    │                                   │                               │
    │                                   │                               │
    │    ┌─────────────────────────────┐│                               │
    │    │         GFW 系统            ││                               │
    │    │                             ││                               │
    │    │  ┌───────┐ ┌───────┐ ┌─────┐│                               │
    │    │  │IP封锁│ │DPI   │ │DNS │ │                               │
    │    │  └───────┘ └───────┘ └─────┘│                               │
    │    │                             │                               │
    │    └─────────────────────────────┘                               │
    │                                   │                               │
    ▼                                   ▼                               ▼
```

GFW 的核心能力：
- **全网流量监控**：部署在主要网络节点
- **实时分析**：高速处理海量流量
- **多层检测**：从 IP 到应用层全覆盖
- **动态更新**：检测规则持续更新

### 检测方法矩阵

GFW 使用多种互补的检测技术：

| 方法 | 层级 | 检测时机 | 特点 | 绕过难度 |
|------|------|----------|------|----------|
| IP 封锁 | 网络层 | 连接建立 | 简单粗暴，易绕过 | 低 |
| DNS 污染 | 应用层 | DNS 查询 | 针对域名解析 | 中 |
| TCP RST | 传输层 | 连接中 | 主动中断 | 中 |
| DPI | 应用层 | 流量传输 | 内容分析 | 高 |
| TLS 指纹 | 应用层 | TLS 握手 | 特征识别 | 高 |
| 主动探测 | 应用层 | 异常检测 | 行为验证 | 最高 |
| 流量统计 | 多层 | 长期观察 | 行为模式 | 高 |

### 审查策略演进

GFW 的检测策略随时间演进：

```
2003-2010: 基础阶段
  │ 主要手段：IP 封锁、关键词过滤
  │ 绕过方式：更换 IP、简单加密
  ▼
2010-2015: 协议识别阶段
  │ 主要手段：DPI 协议识别、DNS 污染
  │ 绕过方式：协议混淆、加密 DNS
  ▼
2015-2020: 深度分析阶段
  │ 主要手段：TLS 指纹、机器学习
  │ 绕过方式：Reality、ShadowTLS
  ▼
2020-至今: 主动探测阶段
  │ 主要手段：主动探测、流量统计
  │ 绕过方式：多层伪装、回落机制
```

## 详解

### IP 封锁

最简单直接的封锁方式。

#### 实现方式

```
+------------------------+
|     IP 封锁机制         |
+------------------------+
          │
    +-----+-----+
    │           │
+--------+  +--------+
|路由黑洞|  |防火墙丢包|
+--------+  +--------+
```

**路由黑洞（BGP层面）**：
- 在路由表中删除目标 IP 的路由
- 所有到该 IP 的流量被丢弃
- 效果：ping 无响应，traceroute 显示断路

**防火墙丢包**：
- 在网络层直接丢弃特定 IP 的包
- 效果：连接超时或拒绝

#### 特征识别

```
从中国境内:
  ping被封IP → 无响应
  traceroute → 在边界处中断
  telnet被封IP:443 → 超时

从境外:
  ping同一IP → 正常响应
  traceroute → 正常路由
```

#### 绕过策略

| 策略 | 方法 | 原理 |
|------|------|------|
| CDN 中转 | 使用 Cloudflare 等 | 代理服务器使用 CDN IP，数量巨大 |
| IP 轮换 | 动态更换服务器 IP | 封锁跟不上更换速度 |
| 域名前置 | Domain Fronting | 连接不同 IP，Host 头为目标域名 |
| 中转节点 | 多层代理 | 通过未封锁的节点中转 |

### DNS 污染

GFW 在 DNS 层面劫持特定域名解析。

#### 工作原理

```
客户端                     GFW                     DNS 服务器
  │                         │                         │
  │── DNS Query (A) ───────→│                         │
  │   QNAME: google.com     │                         │
  │                         │                         │
  │                         │ 检测到敏感域名           │
  │                         │                         │
  │←──伪造 DNS Response ────│                         │
  │   返回错误IP            │                         │
  │                         │                         │
  │                         │── DNS Query ───────────→│
  │                         │←── 正确结果 ────────────│
  │                         │   (被丢弃，不到达客户端) │
```

伪造响应的特点：
- 响应速度比真实响应更快（GFW 就在网络边界）
- 返回固定错误 IP（如国内 IP 或无效 IP）
- 多次查询返回不同错误 IP（增加迷惑性）

#### 特征识别

```bash
# 从中国境内查询
dig google.com @8.8.8.8
# 返回: 93.46.8.89 (错误 IP)

# 从境外查询
dig google.com @8.8.8.8
# 返回: 142.250.x.x (正确 IP)
```

#### 绕过策略

| 策略 | 方法 | 原理 |
|------|------|------|
|加密 DNS | DoH/DoT | DNS 查询加密，GFW 无法识别内容 |
|境外 DNS | 使用境外 DNS 服务器 | 但 DNS 服务器 IP 可能被封锁 |
|硬编码 IP | 客户端直接使用 IP | 绕过 DNS 解析，但 IP 可能被封 |
|DNS over Proxy | DNS 查询走代理 | 完全绕过本地 DNS |

### TCP RST 注入

GFW 主动发送伪造的 TCP RST 包中断连接。

#### 工作原理

```
客户端                     GFW                     目标服务器
  │                         │                         │
  │── TCP SYN ─────────────→│── TCP SYN ─────────────→│
  │←── TCP SYN-ACK ─────────│←── TCP SYN-ACK ────────│
  │── TCP ACK ─────────────→│── TCP ACK ─────────────→│
  │                         │                         │
  │── HTTP GET ───────────→│── HTTP GET ───────────→│
  │   (包含敏感词)          │                         │
  │                         │                         │
  │                         │ 检测到敏感内容           │
  │                         │                         │
  │←──伪造 RST ────────────│                         │
  │←──伪造 RST ─────────────────────────────────────│
  │                         │                         │
  │                         │                         │
连接被双方中断
```

RST 注入的特点：
- 向连接双方各发送一个 RST
- RST 包伪造为对方发送
- 接收方收到后立即断开连接

#### 触发条件

| 触发条件 | 说明 |
|----------|------|
| HTTP敏感词 | HTTP 请求包含敏感关键词 |
| SNI敏感域名 | TLS ClientHello 的 SNI 扩展 |
| 协议特征 | 流量匹配已知代理协议模式 |

#### 绕过策略

| 策略 | 方法 | 原理 |
|------|------|------|
| TLS 加密 | 全链路 TLS | 阻止内容检测 |
| 忽略 RST | 内核层面丢弃 RST | 需要特殊配置，不完全可靠 |
| QUIC | 使用 UDP 协议 | 不受 TCP RST 影响 |

### 深度包检测（DPI）

GFW 最强大的检测手段，分析数据包内容。

#### 检测维度

```
+------------------------+
|     DPI 检测维度        |
+------------------------+
          │
    +-----+-----+
    │           │
+--------+  +----------------+
|协议特征|  |流量统计分析    |
+--------+  +----------------+
    │           │
    │           │
+--------+  +--------+
|握手模式|  |包大小分布|
|认证头  |  |时间间隔  |
|命令格式|  |突发模式  |
+--------+  +--------+
```

#### 协议特征检测

早期代理协议有明显的特征：

| 协议 | 检测特征 | 检测方法 |
|------|----------|----------|
| Shadowsocks (早期) | 固定加密握手模式 | 熵值分析、模式匹配 |
| VMess | 认证头结构 | 固定长度、特定字节分布 |
| OpenVPN | TLS 握手特征 | 特定扩展、证书特征 |
| SSR | 协议混淆特征 | 混淆层可被逆向分析 |

检测示例：

```
Shadowsocks 原始协议特征:
  连接 → 随机16字节 + 加密数据
  熵值接近随机 → 但固定长度可识别
  
检测方法:
  1. 记录连接首包长度
  2. 分析大量连接的长度分布
  3. 固定长度的首包 → Shadowsocks 特征
```

#### 流量统计检测

即使加密，流量元数据仍暴露信息：

```
流量统计分析:
  │
  │ 输入: 大量加密连接的元数据
  │   · 包大小序列
  │   · 时间间隔序列
  │   · 流量方向（上行/下行）
  │   · 持续时间
  │
  ▼
+------------------------+
|    机器学习模型         |
+------------------------+
          │
          ▼
  输出: 流量类型分类
    · 视频流
    · 网页浏览
    · SSH
    ·代理隧道
```

统计特征：

| 特征 | 视频流 | 网页浏览 | 代理隧道 |
|------|--------|----------|----------|
| 包大小分布 | 大包为主 | 小包为主 | 混合 |
| 时间间隔 | 稳定 | 突发 | 可能均匀 |
| 上行/下行比例 | 不对称（下行大） | 短暂不对称 | 可能对称 |
| 持续时间 | 长 | 短 | 可变 |

### TLS 指纹检测

GFW 分析 TLS ClientHello 的指纹特征。

#### JA3/JA4 指纹原理

```
ClientHello 包含:
  │
  │ · TLS 版本
  │ · 密码套件列表（顺序）
  │ · 扩展列表（顺序）
  │ · 椭圆曲线列表
  │ · 点格式列表
  │
  ▼
+------------------------+
|    指纹计算             |
+------------------------+
          │
          ▼
  JA3 = MD5(版本 + 密码套件 + 扩展 + 曲线 + 点格式)
```

不同客户端的指纹差异：

```
Chrome指纹:
  密码套件顺序: AES-128 → AES-256 → ChaCha20
  扩展顺序: 0-5-10-11-13-...
  
Go TLS指纹:
  密码套件顺序: 不同顺序
  扩展顺序: 不同顺序
  
结果: JA3指纹不同
```

#### 指纹不匹配检测

```
客户端发送 ClientHello
    │
    │ SNI: www.microsoft.com
    │ JA3: Go TLS 库特征
    ▼
GFW 分析
    │
    │ 问题:
    │   1. SNI 声称访问 Microsoft
    │   2. 但 JA3 指纹是 Go TLS
    │   3. Microsoft 网站用户用 Chrome/Edge
    │   4. Go TLS 不是浏览器指纹
    │
    ▼
标记为可疑代理流量
```

#### JARM 服务端指纹

JARM 从服务端角度识别服务器：

```
+------------------------+
|   JARM 探测过程         |
+------------------------+
          │
    发送10个特制 ClientHello
          │
    分析服务器响应差异
          │
          ▼
    生成服务器指纹
```

不同 TLS 实现响应不同：
- OpenSSL 响应模式 A
- BoringSSL 响应模式 B
- Go TLS 响应模式 C

代理服务器可能暴露非标准指纹。

#### 绕过策略

| 策略 | 方法 | 原理 |
|------|------|------|
| uTLS | 模拟浏览器指纹 | ClientHello 与真实浏览器一致 |
| Reality | 使用真实网站握手 | 指纹完全来自真实网站 |
| ShadowTLS | 转发握手到真实服务器 | TLS 响应来自真实服务器 |

### 主动探测

GFW 会主动连接可疑服务器验证是否为代理。

#### 探测触发条件

```
流量分析标记可疑
    │
    │ · TLS 指纹异常
    │ · 流量统计异常
    │ · 连接模式异常
    │
    ▼
触发主动探测
```

#### 探测方式

```
+------------------------+
|    主动探测类型         |
+------------------------+
          │
    +-----+-----+
    │           │
+--------+  +--------+
|连接探测|  |协议探测|
+--------+  +--------+
    │           │
    │           │
+--------+  +--------+
|TCP连接 |  |TLS握手 |
|HTTP请求|  |协议命令|
+--------+  +--------+
```

**连接探测**：
- 发起 TCP 连接
- 观察响应行为
- 正常网站 → 返回 HTTP 响应
- 代理服务器 → 可能返回代理响应

**TLS 探测**：
- 完成 TLS 握手
- 检查证书内容
- 正常网站 → 证书与域名匹配
- 代理服务器 → 可能是自签名证书

**协议探测**：
- 发送特定协议命令
- Shadowsocks 命令
- Trojan 凭据
- 观察是否返回代理响应

#### 探测识别逻辑

```
探测连接 ─────────────→ 服务器
                         │
                         │ 响应
                         ▼
+------------------------+
|    响应分析             |
+------------------------+
          │
    +-----+-----+
    │           │
+--------+  +--------+
|正常网站|  |代理服务器|
+--------+  +--------+
    │           │
    │           │
返回HTTP  返回代理协议响应
证书匹配  自签名证书或无证书
无代理特征 代理命令响应

结论:
  正常网站 → 放行
  代理服务器 → 封锁
```

#### 绕过策略

| 策略 | 方法 | 原理 |
|------|------|------|
| Reality | 无 Short ID 时回落 | 探测连接返回真实网站内容 |
| ShadowTLS | 转发探测到真实服务器 | 探测看到的是真实 TLS 服务器 |
| 回落机制 | 非代理流量回落到正常服务 | 正常 HTTP 响应 |

Reality 的探测处理：

```
探测连接 (无 Short ID)
    │
    │ ClientHello
    ▼
Reality服务器
    │
    │ 检测: 无 Short ID 标记
    │ 判断: 这是探测，不是正常客户端
    │
    ▼
转发到真实网站
    │
    │ ←── 返回真实网站响应
    ▼
探测者看到:
  正常 TLS 握手
  正常网站证书
  正常 HTTP 响应
  
结论: 这是个正常网站
```

## 使用示例

### Reality 配置抗探测

```json
{
    "stealth": {
        "reality": {
            "dest": "www.microsoft.com:443",
            "server_names": ["www.microsoft.com"],
            "private_key": "YOUR_PRIVATE_KEY",
            "short_ids": ["abc123", "def456"]
        }
    }
}
```

原理：
- 正常客户端携带 Short ID → 代理服务
- 探测连接无 Short ID → 转发到 Microsoft
- 探测者看到正常 Microsoft 网站

### ShadowTLS 配置

```json
{
    "stealth": {
        "shadowtls": {
            "version": 3,
            "password": "your_password",
            "handshake_dest": "www.microsoft.com:443",
            "strict_mode": true
        }
    }
}
```

原理：
- TLS握手转发到 Microsoft
- 探测看到的是 Microsoft 的 TLS 响应
- 密码验证隐藏在 TLS 内部

### 回落配置

```json
{
    "agent": {
        "camouflage": "/api/v1/stream",
        "reverse_map": {
            "default": "127.0.0.1:8080"
        }
    }
}
```

原理：
- 非代理请求回落到本地 HTTP 服务
- 探测请求得到正常 HTTP 响应
- 只有特定路径才触发代理

## 最佳实践

### 多层防御策略

```
+------------------------+
|    多层防御架构         |
+------------------------+
          │
    第一层: TLS 指纹伪装
          │
          ▼
    第二层: 探测抵抗（Reality/ShadowTLS）
          │
          ▼
    第三层: 回落机制
          │
          ▼
    第四层: 流量混淆
```

### Reality 目标选择原则

| 原则 | 说明 | 推荐目标 |
|------|------|----------|
| TLS 1.3 支持 | 握手兼容 | Microsoft, Apple, Cloudflare |
| 高流量 | 不易异常 | 大型网站 |
| 国外网站 | 不会被主动探测 | 美国科技公司 |
| 指纹标准 | 使用主流 TLS 库 | BoringSSL/OpenSSL |

### 检测规避要点

| 检测方式 | 规避要点 |
|----------|----------|
| IP 封锁 | 使用 CDN、动态 IP |
| DNS 污染 | DoH/DoT、代理 DNS |
| TCP RST | TLS 加密、QUIC |
| DPI | 协议混淆、无特征 |
| TLS 指纹 | uTLS、Reality |
| 主动探测 | 回落机制 |

## 常见问题

### Q1: GFW 能检测所有代理吗？

**A**: 不能，但有不断提高的检测能力：
- 简单协议易被识别
- Reality/ShadowTLS 等新技术暂时安全
- 长期使用同一方式可能被针对
- 需持续更新对抗策略

### Q2: Reality 会被探测识别吗？

**A**: 设计上抵抗探测：
- 探测连接返回真实网站内容
- Short ID 隐藏在 ClientHello 中
- 探测者无法区分正常访问和代理

### Q3: CDN IP 会被封锁吗？

**A**: 可能但困难：
- CDN IP 数量巨大（Cloudflare 有数百万）
- 封锁 CDN IP 影响大量正常网站
- 但特定 CDN IP 可能被针对封锁

### Q4: 如何判断是否被封锁？

**A**: 测试方法：
```bash
# 测试 IP 封锁
ping YOUR_SERVER_IP
traceroute YOUR_SERVER_IP

# 测试 TLS 连接
openssl s_client -connect YOUR_SERVER:443

# 从不同位置测试
# 使用境外 VPS 测试连接
```

### Q5:被封后如何恢复？

**A**: 策略：
- 更换服务器 IP
- 更换域名
- 更换伪装目标
- 使用 CDN 中转
- 避免使用被针对的方式

## 排障指南

### 问题：连接突然中断

**症状**: 正常使用中连接断开

**排查步骤**:

1. 检查是否被封锁
   ```bash
   ping YOUR_SERVER
   ```

2. 检查是否被 RST
   - 使用Wireshark 抓包
   - 查看 TCP RST 包

3. 检查服务器状态
   - 服务器是否正常运行
   - 配置是否正确

---

### 问题：DNS 解析失败

**症状**: 无法解析域名

**排查步骤**:

1. 测试 DNS 污染
   ```bash
   dig YOUR_DOMAIN @8.8.8.8
   dig YOUR_DOMAIN @dns.google
   ```

2. 使用加密 DNS
   ```bash
   # DoH
   curl "https://dns.google/resolve?name=YOUR_DOMAIN"
   ```

3. 检查代理 DNS 配置
   - 确保DNS 查询走代理

### 问题：TLS 指纹被标记

**症状**: 连接建立但立即中断

**排查步骤**:

1. 检查 TLS 指纹
   ```bash
   ja3 YOUR_SERVER:443
   ```

2. 检查 SNI 与指纹匹配
   - SNI 声称的网站
   - 实际 TLS 指纹类型

3. 更换伪装方案
   - 使用 Reality
   - 使用 ShadowTLS

---

### 问题：主动探测确认封锁

**症状**: 服务器被封锁，疑似探测

**排查步骤**:

1. 分析探测特征
   - 探测 IP 来源
   - 探测时间模式

2. 检查回落机制
   - 回落是否正常工作
   - 探测看到什么

3. 重新配置
   - 更换伪装目标
   - 更换协议
